# 整合測試品質評估器（ITQA）

用於評估 ASP.NET MVC 專案整合測試套件的系統化評估框架。此技能提供決策樹、品質標準，以及在不損害產品程式碼的前提下診斷和修正測試失敗的成熟模式。

## 核心目的

將臨時性的測試疑難排解轉化為可重複執行、架構驗證過的方法論。面對測試失敗時，ITQA 提供結構化的根本原因分析與修正優先序。

## 何時使用此技能

**主要情境：**
- 評估整合測試套件的健康狀態（冒煙測試、特徵測試、對應基準測試）
- 在 ASP.NET MVC 4.x 專案中除錯測試失敗
- 評估測試能否在 CI/CD 環境中執行（vstest.console.exe、Visual Studio Test Explorer）
- 判斷測試失敗需要修改產品程式碼，還是可以在測試層解決

**次要情境：**
- 建立測試驗收的品質閘門（通過率門檻、已知限制文件）
- 為新測試套件建立可複製的測試模式
- 記錄測試基礎設施的架構決策

## 快速入門：三階段評估

### 第一階段：基準線（15 分鐘）
1. 執行完整測試套件：`vstest.console.exe [test-assembly] /Logger:trx`
2. 擷取指標：總數、通過數、失敗數、略過數
3. 計算通過率：`(通過 + 略過) / 總數`
4. **閘門**：通過率 ≥95% → 進入第二階段。<95% → 進入診斷流程

### 第二階段：診斷（30-60 分鐘）
1. 對每個失敗測試單獨執行，以隔離根本原因
2. 將失敗分類：基礎設施 | 資料 | 邏輯 | 產品缺陷
3. 對每個類別，參考[決策框架](references/decision-framework.md)
4. 優先提出測試層修正方案，並以 ≥1 次執行驗證
5. 唯有在測試層無法解決時才升級至產品程式碼

### 第三階段：驗證（30 分鐘）
1. 實作所有提議的修正方案
2. 獨立執行完整套件 ≥3 次
3. 驗證可重現性（各次執行的通過率相同）
4. 執行[品質檢查清單](references/quality-checklist.md)
5. 產生含根本原因記錄的評估報告

## 決策框架快速參考

完整決策樹詳見[決策框架](references/decision-framework.md)。快速參考：

| 失敗類別 | 根本原因模式 | 測試層修正 | 產品程式碼風險 |
|---|---|---|---|
| **基礎設施** | MapPath 為 null、HttpContext 遺失 | Try-catch 降級、環境防護 | ✅ 低（僅被動） |
| **資料** | 測試身份不在 DB 中、設定表格缺失 | DB 驅動解析器、動態探索 | ✅ 低（僅查詢） |
| **邏輯** | 業務規則中的例外狀況、無效狀態 | 測試情境覆寫、Mock 工廠 | ✅ 低（僅測試層） |
| **產品缺陷** | 型別不符（NHibernate）、Schema 漂移 | SQL 降級（臨時）、標記到 Sprint | ⚠️ 中（未來修正） |

## 品質檢查清單

所有評估必須通過此檢查清單。詳細驗證步驟請見[品質檢查清單](references/quality-checklist.md)。

- [ ] **通過率**：≥95%（或對每個失敗類別有明確記錄的例外情況）
- [ ] **產品程式碼影響**：0 個行為變更（僅 try-catch，無分支）
- [ ] **可重現性**：在 ≥3 次獨立測試執行中結果相同
- [ ] **根本原因**：每個失敗均已識別至層次（基礎設施／資料／邏輯／缺陷）
- [ ] **已知問題**：所有限制均已明確記錄（不以 Ignore/Skip 隱藏）
- [ ] **維護負擔**：測試程式碼修改量 ≤ 觸及檔案的 20%
- [ ] **文件**：架構決策、降級模式、升級標準均已記錄

## OnlineService_SG 評估案例的關鍵模式

此技能源自對 OnlineService_SG 專案的成功評估（153 個整合測試 → 151 個通過，2 個已知限制）。

### 模式一：Try-Catch 優雅降級
針對在測試執行器中失敗的環境特定 API（例如 `HttpContext.Current.Server.MapPath`）：

```csharp
try {
    path = HttpContext.Current?.Server.MapPath(virtualPath);
}
catch {
    path = ResolvePath(virtualPath);  // 降級為與環境無關的方法
}
```

✅ 優點：單一程式碼路徑同時適用於生產和測試環境  
✅ 優點：不變更產品程式碼行為（降級是靜默的）  
⚠️ 限制：降級方案必須在功能上等效  

**使用時機**：4 個以上的測試在同一個環境限制上失敗

### 模式二：DB 驅動的身份探索
針對需要有效資料庫帳號的測試（例如分支設定檔、使用者角色）：

1. 查詢 SETUP_PERSONAL_ROLE 和相關表格，找出符合所需條件的帳號
2. 動態提取 roleOuCode、accountCode
3. 用探索到的值取代硬編碼的測試身份
4. 驗證目標環境中至少存在 1 個帳號

✅ 優點：測試能適應實際資料庫內容  
✅ 優點：消除硬編碼測試資料（脆弱性來源）  
✅ 優點：早期偵測遺失的設定資料  

**使用時機**：3 個以上的測試因「找不到帳號」、「無效憑證」而失敗

### 模式三：託管 MVC 鷹架模擬
針對需要具備適當路由的合成 HttpContext 的控制器層測試：

```csharp
ReadOnlyHttpContextFactory.SetupHttpContext(
    ReadOnlyIdentityFactory.CreateBranchPrincipal(roleOuCode));
var controller = new YourController { 
    ControllerContext = CreateControllerContext() 
};
```

✅ 優點：模擬 BaseController 初始化、路由、Principal 注入  
✅ 優點：將控制器邏輯與託管環境隔離  

**使用時機**：5 個以上的測試在控制器實例化或 User.Identity 屬性上失敗

### 模式四：實體對應缺陷的 SQL 降級
當 NHibernate 實體實例化失敗（型別不符）時，使用原始 SQL：

```csharp
// 取代：session.CreateCriteria(typeof(Entity)).List<Entity>()
// 使用：session.CreateSQLQuery("SELECT * FROM TABLE LIMIT 1").List()
```

✅ 優點：測試得以執行，對應缺陷得以浮現（而非隱藏）  
✅ 優點：保留測試覆蓋訊號，同時記錄產品問題  
⚠️ 限制：降低對應驗證強度（僅為臨時權宜措施）  

**使用時機**：1-2 個測試因型別轉換失敗（記錄為已知限制）

## 升級標準

**升級至架構團隊的情況：**
- 需要修改產品程式碼行為（而非只加 try-catch）
- 型別／對應不符影響超過 5% 的實體
- 環境限制在測試層無法滿足
- 需要多個基礎設施變通方案（顯示存在更深的架構債務）

**升級至產品工程的情況：**
- NHibernate 對應缺陷（型別不符、Schema 漂移）
- 需要變更資料庫 Schema
- 產品程式碼中的 API 合約違規
- 測試執行期間偵測到效能衰退

**無需升級（測試層已足夠）的情況：**
- Try-catch 降級能處理環境限制
- DB 驅動解析器能探索有效的測試身份
- Mock／覆寫能隔離測試邏輯層
- SQL 降級是可接受的臨時措施（附文件說明）

## 實作說明

針對新專案評估：

1. **設定（5 分鐘）**：若框架為 ASP.NET MVC 4.x，從 OnlineService_SG 複製測試基礎設施
   - 複製 `ReadOnlyHttpContextFactory.cs`
   - 複製 `ReadOnlyIdentityFactory.cs`
   - 複製 `ReadOnlyDbEnvironmentGuard.cs`

2. **執行基準線（5 分鐘）**：執行一次完整套件，擷取通過／失敗數量

3. **診斷（30-60 分鐘）**：使用決策框架對每個失敗進行分類

4. **修正（15-45 分鐘）**：依優先序實作修正（基礎設施 → 資料 → 邏輯 → 產品缺陷）

5. **驗證（30 分鐘）**：執行套件 ≥3 次，確認可重現性

6. **報告（15 分鐘）**：填寫執行摘要模板，存檔調查結果

**預計工時**：具備 ≥90% 基準通過率的 100-200 個測試套件，需 2-3 小時
