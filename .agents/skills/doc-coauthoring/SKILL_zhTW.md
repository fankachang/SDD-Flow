# 文件協作撰寫工作流程

本技能提供結構化的工作流程，引導使用者完成協作式文件建立。主動扮演引導者的角色，帶領使用者完成三個階段：脈絡蒐集、精煉與結構化，以及讀者測試。

## 何時提供此工作流程

**觸發條件：**
- 使用者提到撰寫文件：「寫一份文件」、「草擬提案」、「建立規格」、「寫報告」
- 使用者提到特定文件類型：「PRD」、「設計文件」、「決策文件」、「RFC」
- 使用者似乎正在開始一項實質性的寫作任務

**初始提案：**
向使用者提供協作撰寫文件的結構化工作流程。說明三個階段：

1. **脈絡蒐集**：使用者提供所有相關脈絡，Claude 提出澄清問題
2. **精煉與結構化**：透過腦力激盪和編輯，逐段建立文件
3. **讀者測試**：用全新的 Claude（無上下文）測試文件，在他人閱讀前找出盲點

說明此方法有助於確保文件在他人閱讀時（包括將文件貼給 Claude 時）效果良好。詢問他們是否想嘗試此工作流程，或偏好自由發揮。

若使用者拒絕，以自由形式工作。若接受，進入第一階段。

## 第一階段：脈絡蒐集

**目標：** 縮短使用者所知與 Claude 所知之間的差距，以便後續提供有智慧的引導。

### 初始問題

首先詢問使用者關於文件的元脈絡：

1. 這是什麼類型的文件？（例如技術規格、決策文件、提案）
2. 主要對象是誰？
3. 有人閱讀這份文件後，期望達到的效果是什麼？
4. 是否有要遵循的範本或特定格式？
5. 還有其他限制或需要了解的脈絡嗎？

告知他們可以用簡短方式回答，或以任何方便的方式傾倒資訊。

**若使用者提供範本或提到文件類型：**
- 詢問他們是否有要共享的範本文件
- 若提供了共享文件的連結，使用適當的整合功能取得文件
- 若提供了檔案，讀取內容

**若使用者提到要編輯現有的共享文件：**
- 使用適當的整合功能讀取目前狀態
- 檢查沒有替代文字的圖片
- 若圖片沒有替代文字，解釋當他人使用 Claude 理解文件時，Claude 將無法看到這些圖片。詢問是否要生成替代文字。若是，請他們將每張圖片貼入對話以生成描述性替代文字。

### 資訊傾倒

初始問題回答後，鼓勵使用者傾倒所有脈絡。請求如下資訊：
- 專案/問題的背景
- 相關的團隊討論或共享文件
- 為何不使用替代方案
- 組織脈絡（團隊動態、過去事件、政治因素）
- 時間壓力或限制
- 技術架構或相依性
- 利害關係人的顧慮

建議他們不用擔心整理——只要把所有東西說出來。提供多種提供脈絡的方式：
- 意識流傾倒
- 指向團隊頻道或討論串
- 連結至共享文件

**若有整合功能可用**（例如 Slack、Teams、Google Drive、SharePoint 或其他 MCP 伺服器），提及可直接用來取得脈絡。

**若在 Claude.ai 或 Claude app 中未偵測到整合功能：** 建議他們可在 Claude 設定中啟用連接器，讓系統直接從訊息應用程式和文件儲存中取得脈絡。

告知他們在完成初始傾倒後，會再提出澄清問題。

**脈絡蒐集期間：**

- 若使用者提到團隊頻道或共享文件：
  - 若有整合功能：告知將立即讀取內容，然後使用適當的整合功能
  - 若無整合功能：說明無法存取。建議在 Claude 設定中啟用連接器，或直接貼上相關內容。

- 若使用者提到不熟悉的實體/專案：
  - 詢問是否應搜尋連接的工具以獲取更多資訊
  - 等待使用者確認後再搜尋

- 使用者提供脈絡時，追蹤已了解的內容和尚不清楚的部分

**提出澄清問題：**

當使用者示意已完成初始傾倒（或在提供大量脈絡後），提出澄清問題以確保理解：

根據脈絡的空白，生成 5-10 個有編號的問題。

告知他們可用簡短方式回答（例如「1: 是的，2: 見 #channel，3: 不行，因為向後相容性」）、連結更多文件、指向頻道，或繼續傾倒資訊——選擇對他們最有效率的方式。

**退出條件：**
當問題顯示已充分理解脈絡時，即可進入下一階段——能夠詢問邊緣案例和取捨，而無需解釋基礎知識。

**過渡：**
詢問是否還有其他脈絡要提供，或是否可以進入文件草稿階段。

若使用者想補充，讓他們繼續。準備好後，進入第二階段。

## 第二階段：精煉與結構化

**目標：** 透過腦力激盪、篩選和迭代精煉，逐段建立文件。

**給使用者的說明：**
說明將逐段建立文件。每一段：
1. 提出澄清問題，確認要包含的內容
2. 腦力激盪 5-20 個選項
3. 使用者指出要保留/移除/合併的項目
4. 草擬該段內容
5. 透過精準修改進行精煉

從最有未知數的段落開始（通常是核心決策/提案），再處理其他部分。

**段落順序：**

若文件結構清晰：
詢問要從哪個段落開始。

建議從最有未知數的段落開始。對決策文件來說，通常是核心提案；對規格文件來說，通常是技術方案。摘要段落最好留到最後。

若使用者不知道需要哪些段落：
根據文件類型和範本，建議 3-5 個適合的段落。

詢問這個結構是否可行，或是否要調整。

**確定結構後：**

建立含所有段落占位符文字的初始文件結構。

**若可使用 artifacts：**
使用 `create_file` 建立 artifact。這給 Claude 和使用者一個可共同參照的骨架。

告知將建立含所有段落標題和簡短占位符文字的初始結構。

建立含所有段落標題和占位符文字（如「[待撰寫]」或「[內容於此]」）的 artifact。

提供 artifact 連結並說明準備開始填寫每個段落。

**若無法使用 artifacts：**
在工作目錄中建立 Markdown 檔案。適當命名（例如 `decision-doc.md`、`technical-spec.md`）。

告知將建立含所有段落標題和占位符文字的初始結構。

建立含所有段落標題和占位符文字的檔案。

確認已建立該檔案名稱，並說明準備開始填寫每個段落。

**每個段落的處理：**

### 步驟一：澄清問題

宣告將開始處理「[段落名稱]」段落。針對應包含的內容提出 5-10 個澄清問題：

根據脈絡和段落目的，生成 5-10 個具體問題。

告知他們可用簡短方式回答，或直接說明重要的內容。

### 步驟二：腦力激盪

對「[段落名稱]」段落，根據段落複雜度，腦力激盪 [5-20] 個可能包含的項目。尋找：
- 曾分享但可能被遺忘的脈絡
- 尚未提到的角度或考量

根據段落複雜度，生成 5-20 個有編號的選項。最後提供腦力激盪更多選項的可能。

### 步驟三：篩選

詢問哪些要保留、移除或合併。請求簡短的理由，以便了解後續段落的優先順序。

提供範例：
- 「保留 1、4、7、9」
- 「移除 3（與 1 重複）」
- 「移除 6（讀者已知道這些）」
- 「合併 11 和 12」

**若使用者給出自由形式的回饋**（例如「看起來不錯」或「大部分我喜歡，但……」）而非選號，提取他們的偏好並繼續。解析他們想保留/移除/更改的內容並執行。

### 步驟四：缺口檢查

根據他們選擇的內容，詢問「[段落名稱]」段落是否有遺漏的重要內容。

### 步驟五：草稿撰寫

使用 `str_replace` 將此段落的占位符文字替換為實際草稿內容。

宣告將根據他們選擇的內容草稿「[段落名稱]」段落。

**若使用 artifacts：**
草稿完成後，提供 artifact 連結。

請他們仔細閱讀，並說明要修改的地方。說明越具體，有助於了解他們的偏好以便處理後續段落。

**若使用檔案（無 artifacts）：**
草稿完成後，確認完成。

告知「[段落名稱]」段落已在 [檔名] 中完成草稿。請他們仔細閱讀，並說明要修改的地方。

**給使用者的重要說明（草稿第一個段落時提醒）：**
提示：不要直接編輯文件，而是說明要修改的地方。這有助於了解他們的風格，以便處理後續段落。例如：「移除 X 項目——Y 已涵蓋了」或「讓第三段更簡潔」。

### 步驟六：迭代精煉

使用者提供回饋時：
- 使用 `str_replace` 進行修改（絕不重新列印整份文件）
- **若使用 artifacts：** 每次修改後提供 artifact 連結
- **若使用檔案：** 只確認修改完成
- 若使用者直接編輯文件後請求讀取：留意他們所做的修改，並在後續段落中牢記（這反映了他們的偏好）

**持續迭代**直到使用者對該段落滿意。

### 品質檢查

連續 3 次迭代沒有實質性修改後，詢問是否有任何內容可以移除而不失去重要資訊。

段落完成後，確認「[段落名稱]」已完成。詢問是否準備好進入下一個段落。

**對所有段落重複此流程。**

### 接近完成時

當接近完成（80% 以上的段落完成），宣告要重新閱讀整份文件並檢查：
- 各段落之間的流暢度和一致性
- 冗餘或矛盾之處
- 任何感覺像「填充內容」或通用廢話的部分
- 每句話是否都有其份量

閱讀整份文件並提供回饋。

**所有段落草稿和精煉完成後：**
宣告所有段落均已完成草稿。說明將最後審閱整份文件。

審閱整體連貫性、流暢度和完整性。

提供任何最終建議。

詢問是否準備好進入讀者測試，或是否還想精煉某些部分。

## 第三階段：讀者測試

**目標：** 用全新的 Claude（無上下文殘留）測試文件，驗證它對讀者的效果。

**給使用者的說明：**
說明現在要測試文件是否對讀者真正有效。這能找出盲點——對作者來說清楚的東西，對其他人可能造成困惑。

### 測試方法

**若可使用子 agent（例如在 Claude Code 中）：**

直接進行測試，無需使用者參與。

### 步驟一：預測讀者問題

宣告將預測讀者閱讀文件時可能提出的問題。

生成 5-10 個讀者實際上可能提問的問題。

### 步驟二：用子 Agent 測試

宣告將用全新的 Claude 實例（無來自本對話的上下文）測試這些問題。

對每個問題，以僅含文件內容和問題的方式呼叫子 agent。

總結每個問題中讀者 Claude 答對/答錯的地方。

### 步驟三：執行額外檢查

宣告將執行額外檢查。

呼叫子 agent 檢查歧義、錯誤假設、矛盾。

總結發現的任何問題。

### 步驟四：回報並修正

若發現問題：
回報讀者 Claude 在特定問題上遇到困難。

列出具體問題。

說明將修正這些缺口。

回到有問題的段落進行精煉。

---

**若無法使用子 agent（例如 claude.ai 網頁介面）：**

使用者需要手動進行測試。

### 步驟一：預測讀者問題

詢問人們在找這份文件時可能會問什麼問題。他們會在 Claude.ai 中輸入什麼？

生成 5-10 個讀者實際上可能提問的問題。

### 步驟二：設定測試

提供測試說明：
1. 開啟全新的 Claude 對話：https://claude.ai
2. 貼上或分享文件內容（若使用支援連接器的共享文件平台，提供連結）
3. 向讀者 Claude 提出生成的問題

對每個問題，指示讀者 Claude 提供：
- 答案
- 任何不清楚或有歧義的地方
- 文件假設讀者已知的知識/脈絡

檢查讀者 Claude 是否給出正確答案或有所誤解。

### 步驟三：額外檢查

也詢問讀者 Claude：
- 「這份文件中哪些內容對讀者來說可能不清楚或有歧義？」
- 「這份文件假設讀者已具備哪些知識或脈絡？」
- 「是否有任何內部矛盾或不一致？」

### 步驟四：根據結果迭代

詢問讀者 Claude 答錯了什麼或遇到困難的地方。說明將修正這些缺口。

回到有問題的段落進行精煉。

---

### 退出條件（兩種方法皆適用）

當讀者 Claude 能一致地正確回答問題，且不再出現新的缺口或歧義時，文件即已完成。

## 最終審閱

讀者測試通過後：
宣告文件已通過讀者 Claude 測試。在完成前：

1. 建議他們自行做最後閱讀——他們擁有這份文件，需要為其品質負責
2. 建議再次確認任何事實、連結或技術細節
3. 請他們確認文件達到了預期的效果

詢問是否需要最後一次審閱，或是否已完成。

**若使用者想要最後審閱，提供審閱。否則：**
宣告文件完成。提供幾個最終提示：
- 考慮在附錄中連結此對話，讓讀者了解文件的開發過程
- 使用附錄提供深度，而不讓主文件過於臃腫
- 在收到真實讀者的回饋後更新文件

## 有效引導的技巧

**語氣：**
- 直接且程序化
- 在影響使用者行為時簡短解釋原因
- 不要「推銷」這個方法——直接執行

**處理偏離：**
- 若使用者想跳過某個階段：詢問是否要跳過並自由發揮
- 若使用者似乎感到不耐煩：承認這比預期花更多時間。建議加快進行的方式
- 始終給使用者調整流程的主導權

**上下文管理：**
- 自始至終，若提到的內容有脈絡缺口，主動詢問
- 不要讓缺口累積——隨時出現隨時處理

**Artifact 管理：**
- 使用 `create_file` 草擬完整段落
- 所有編輯使用 `str_replace`
- 每次修改後提供 artifact 連結
- 腦力激盪列表不用 artifact——那只是對話

**品質勝於速度：**
- 不要急於完成各階段
- 每次迭代應帶來有意義的改善
- 目標是一份對讀者真正有效的文件
