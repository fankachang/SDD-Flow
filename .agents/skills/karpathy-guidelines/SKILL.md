# Karpathy Guidelines

源自 Andrej Karpathy 對 LLM 編碼缺陷的觀察，四條原則直接應對常見問題。

**權衡說明：** 這些準則偏向謹慎而非速度。對於瑣碎任務，請自行判斷。

---

## 1. 先思考再編碼

**不要假設。不要隱藏困惑。提出取捨。**

實作前：
- 明確陳述你的假設。若不確定，就提問。
- 若存在多種解讀，逐一呈現——不要默默挑選。
- 若有更簡單的方式，說出來。適時提出異議。
- 若有不清楚之處，停下來。說明困惑點，提問。

---

## 2. 簡單優先

**以最少的程式碼解決問題。不做推測性的實作。**

- 不加入使用者未要求的功能。
- 不為一次性使用的程式碼建立抽象層。
- 不加入未被要求的「彈性」或「可配置性」。
- 不為不可能發生的情境加入錯誤處理。
- 若寫了 200 行而 50 行即可達成，重寫它。

自我測試：「資深工程師會說這過度複雜嗎？」若是，簡化。

---

## 3. 精準變更

**只碰必須碰的。只清理自己造成的混亂。**

編輯現有程式碼時：
- 不「改善」相鄰的程式碼、註解或格式。
- 不重構沒有損壞的東西。
- 沿用現有風格，即使你會換一種寫法。
- 若發現不相關的死碼，提及它——但不要刪除。

當你的變更產生孤兒（orphans）時：
- 移除「你的變更」所造成的未使用 import／變數／函式。
- 不移除原本已存在的死碼，除非被要求。

自我測試：每一行變更都應能直接追溯到使用者的要求。

---

## 4. 目標驅動執行

**定義成功標準。循環直到驗證通過。**

將任務轉化為可驗證的目標：
- 「新增驗證」→「為無效輸入撰寫測試，再讓測試通過」
- 「修復 bug」→「撰寫能重現 bug 的測試，再讓測試通過」
- 「重構 X」→「確保重構前後測試均通過」

多步驟任務時，先陳述簡短計畫：
```
1. [步驟] → 驗證：[檢查項目]
2. [步驟] → 驗證：[檢查項目]
3. [步驟] → 驗證：[檢查項目]
```

明確的成功標準讓 LLM 可以獨立循環執行。模糊的標準（「讓它能動」）需要不斷釐清。

---

**這些準則奏效的跡象：**
- diff 中更少不必要的變更
- 因過度複雜而需重寫的次數減少
- 釐清問題的提問在實作前發生，而非在出錯後
