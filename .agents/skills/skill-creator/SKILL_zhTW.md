# 技能建立工具

本技能提供建立有效技能的指引。

## 關於技能

技能是模組化、自包含的套件，透過提供專業知識、工作流程和工具來擴展 Claude 的能力。可以將它們視為特定領域或任務的「入職指南」——它們將 Claude 從通用型 agent 轉變為配備特定程序知識的專業 agent，而這些知識是任何模型都無法完全具備的。

### 技能提供的內容

1. 專業工作流程——特定領域的多步驟流程
2. 工具整合——使用特定檔案格式或 API 的說明
3. 領域專業知識——公司特有的知識、Schema、業務邏輯
4. 捆綁資源——用於複雜和重複性任務的腳本、參考資料和素材

## 核心原則

### 簡潔是關鍵

上下文視窗是公共資源。技能與 Claude 所需的其他一切共享上下文視窗：系統提示、對話歷史、其他技能的中繼資料和實際的使用者請求。

**預設假設：Claude 本身已非常聰明。** 只需新增 Claude 尚未具備的上下文。對每條資訊提出質疑：「Claude 真的需要這個解釋嗎？」和「這段內容值得佔用的 token 嗎？」

優先使用簡潔的範例，而非冗長的解說。

### 設定適當的自由度

將具體程度與任務的脆弱性和可變性匹配：

**高自由度（文字指示）**：適用於多種方法皆可行、決策取決於上下文，或以啟發式方法引導的情況。

**中等自由度（帶參數的虛擬碼或腳本）**：適用於存在偏好模式、允許一定變化，或設定會影響行為的情況。

**低自由度（特定腳本、少量參數）**：適用於操作容易出錯、一致性至關重要，或必須遵循特定順序的情況。

可以想像 Claude 在探索路徑：懸崖旁的窄橋需要具體的護欄（低自由度），而開闊的田野則允許多種路線（高自由度）。

### 技能的結構

每個技能由一個必要的 SKILL.md 檔案和可選的捆綁資源組成：

```
skill-name/
├── SKILL.md（必要）
│   ├── YAML frontmatter 中繼資料（必要）
│   │   ├── name:（必要）
│   │   └── description:（必要）
│   └── Markdown 指示（必要）
└── 捆綁資源（可選）
    ├── scripts/          — 可執行程式碼（Python/Bash 等）
    ├── references/       — 視需要載入上下文的文件
    └── assets/           — 用於輸出的檔案（範本、圖示、字型等）
```

#### SKILL.md（必要）

每個 SKILL.md 包含：

- **Frontmatter**（YAML）：包含 `name` 和 `description` 欄位。這些是 Claude 用來判斷何時使用技能的唯一欄位，因此在描述技能內容及使用時機時必須清楚且全面。
- **Body**（Markdown）：使用技能的指示和指引。只有在技能觸發後才會載入。

#### 捆綁資源（可選）

##### 腳本（`scripts/`）

用於需要確定性可靠度或會重複撰寫的任務的可執行程式碼（Python/Bash 等）。

- **何時包含**：當相同程式碼被反覆重寫，或需要確定性可靠度時
- **範例**：用於 PDF 旋轉任務的 `scripts/rotate_pdf.py`
- **優點**：節省 token、確定性強，可在不載入上下文的情況下執行
- **注意**：腳本仍可能需要 Claude 讀取，以便進行修補或環境特定調整

##### 參考資料（`references/`）

文件和參考資料，視需要載入上下文，以告知 Claude 的思考和處理過程。

- **何時包含**：當 Claude 在工作時應參考的文件
- **範例**：財務 Schema 的 `references/finance.md`、公司 NDA 範本的 `references/mnda.md`、公司政策的 `references/policies.md`、API 規格的 `references/api_docs.md`
- **使用場景**：資料庫 Schema、API 文件、領域知識、公司政策、詳細工作流程指南
- **優點**：保持 SKILL.md 精簡，只在 Claude 判斷需要時才載入
- **最佳實踐**：若檔案很大（超過 10,000 字），在 SKILL.md 中包含 grep 搜尋模式
- **避免重複**：資訊應存在於 SKILL.md 或參考檔案中的其中一個，而非兩者都有。對於詳細資訊，優先使用參考檔案——這讓 SKILL.md 保持精簡，同時讓資訊可被發現，而不佔用上下文視窗。SKILL.md 只保留必要的程序指示和工作流程指南；詳細參考資料、Schema 和範例移至參考檔案。

##### 素材（`assets/`）

不打算載入上下文，而是用在 Claude 產出的輸出中的檔案。

- **何時包含**：技能需要用於最終輸出的檔案時
- **範例**：品牌素材的 `assets/logo.png`、PowerPoint 範本的 `assets/slides.pptx`、HTML/React 樣板的 `assets/frontend-template/`、字型的 `assets/font.ttf`
- **使用場景**：範本、圖片、圖示、樣板程式碼、字型、會被複製或修改的範例文件
- **優點**：將輸出資源與文件分離，讓 Claude 能使用檔案而無需載入上下文

#### 不應包含在技能中的內容

技能應只包含直接支援其功能的必要檔案。**不要**建立無關文件或輔助檔案，包括：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 等等

技能應只包含 AI agent 完成任務所需的資訊。不應包含關於建立過程的輔助說明、設定和測試流程、面向使用者的文件等。建立額外的文件檔案只會增加雜亂和混淆。

### 漸進式揭露設計原則

技能使用三層載入系統來有效管理上下文：

1. **中繼資料（name + description）** — 始終在上下文中（約 100 字）
2. **SKILL.md 本文** — 技能觸發時（不超過 5,000 字）
3. **捆綁資源** — 由 Claude 視需要載入（無限制，因為腳本無需讀入上下文即可執行）

#### 漸進式揭露模式

SKILL.md 本文保持在必要內容以下，500 行以內，以減少上下文膨脹。接近此限制時，將內容拆分到獨立檔案。拆分時，必須在 SKILL.md 中參照這些檔案，並清楚說明何時讀取，確保技能讀者知道它們的存在以及使用時機。

**關鍵原則：** 當技能支援多種變體、框架或選項時，SKILL.md 只保留核心工作流程和選擇指引。將變體特定的細節（模式、範例、設定）移至獨立的參考檔案。

**模式一：高層次指南搭配參考資料**

```markdown
# PDF 處理

## 快速入門

使用 pdfplumber 擷取文字：
[程式碼範例]

## 進階功能

- **表單填寫**：完整指南見 [FORMS.md](FORMS.md)
- **API 參考**：所有方法見 [REFERENCE.md](REFERENCE.md)
- **範例**：常見模式見 [EXAMPLES.md](EXAMPLES.md)
```

Claude 只在需要時才載入 FORMS.md、REFERENCE.md 或 EXAMPLES.md。

**模式二：按領域組織**

對於支援多個領域的技能，按領域組織內容以避免載入不相關的上下文：

```
bigquery-skill/
├── SKILL.md（概覽與導覽）
└── reference/
    ├── finance.md（收入、帳務指標）
    ├── sales.md（機會、管道）
    ├── product.md（API 使用量、功能）
    └── marketing.md（活動、歸因）
```

使用者詢問銷售指標時，Claude 只需讀取 sales.md。

同樣，對於支援多個框架或變體的技能，按變體組織：

```
cloud-deploy/
├── SKILL.md（工作流程＋供應商選擇）
└── references/
    ├── aws.md（AWS 部署模式）
    ├── gcp.md（GCP 部署模式）
    └── azure.md（Azure 部署模式）
```

使用者選擇 AWS 時，Claude 只需讀取 aws.md。

**模式三：條件式細節**

展示基本內容，連結進階內容：

```markdown
# DOCX 處理

## 建立文件

使用 docx-js 建立新文件。見 [DOCX-JS.md](DOCX-JS.md)。

## 編輯文件

簡單編輯可直接修改 XML。

**追蹤變更**：見 [REDLINING.md](REDLINING.md)
**OOXML 細節**：見 [OOXML.md](OOXML.md)
```

使用者需要這些功能時，Claude 才讀取 REDLINING.md 或 OOXML.md。

**重要指南：**

- **避免深層巢狀參照** — 參照保持在 SKILL.md 的一層深度以內。所有參考檔案應直接從 SKILL.md 連結。
- **較長的參考檔案需有結構** — 超過 100 行的檔案，頂部加上目錄，讓 Claude 在預覽時能掌握全貌。

## 技能建立流程

技能建立包含以下步驟：

1. 透過具體範例理解技能
2. 規劃可重用的技能內容（腳本、參考資料、素材）
3. 初始化技能（執行 init_skill.py）
4. 編輯技能（實作資源並撰寫 SKILL.md）
5. 打包技能（執行 package_skill.py）
6. 根據實際使用情況迭代改善

按順序執行這些步驟，只有在有明確理由時才跳過。

### 步驟一：透過具體範例理解技能

只有在技能的使用模式已清楚理解時才跳過此步驟。即使在處理現有技能時，此步驟仍有其價值。

要建立有效的技能，必須清楚理解技能將被使用的具體範例。這些理解可來自使用者提供的直接範例，或由你生成後請使用者確認的範例。

例如，建立圖片編輯技能時，相關問題包括：

- 「圖片編輯技能應支援哪些功能？編輯、旋轉，還有其他嗎？」
- 「你能舉些使用這個技能的例子嗎？」
- 「我想像使用者可能會說：『移除這張圖片的紅眼』或『旋轉這張圖片』。你還能想到其他使用方式嗎？」
- 「使用者說什麼應該觸發這個技能？」

為避免讓使用者感到不知所措，避免在一則訊息中問太多問題。從最重要的問題開始，視需要再跟進。

對技能應支援的功能有清楚認識後，即可結束此步驟。

### 步驟二：規劃可重用的技能內容

要將具體範例轉化為有效的技能，分析每個範例的方式如下：

1. 考慮如何從頭執行這個範例
2. 識別在重複執行這些工作流程時，哪些腳本、參考資料和素材會有幫助

範例：建立 `pdf-editor` 技能處理「幫我旋轉這個 PDF」的請求時，分析顯示：

1. 旋轉 PDF 每次都需要重新撰寫相同程式碼
2. 在技能中儲存 `scripts/rotate_pdf.py` 腳本會很有幫助

範例：設計 `frontend-webapp-builder` 技能處理「幫我做一個待辦清單 app」或「幫我做一個追蹤步數的 dashboard」時，分析顯示：

1. 撰寫前端 webapp 每次都需要相同的樣板 HTML/React
2. 在技能中儲存包含樣板 HTML/React 專案檔案的 `assets/hello-world/` 範本會很有幫助

範例：建立 `big-query` 技能處理「今天有多少使用者登入？」時，分析顯示：

1. 查詢 BigQuery 每次都需要重新發現資料表 Schema 和關聯
2. 在技能中儲存記錄資料表 Schema 的 `references/schema.md` 檔案會很有幫助

要確立技能的內容，分析每個具體範例，建立要包含的可重用資源清單：腳本、參考資料和素材。

### 步驟三：初始化技能

現在是實際建立技能的時候了。

只有在要開發的技能已存在，且需要迭代或打包時才跳過此步驟。這種情況下，繼續到下一個步驟。

從頭建立新技能時，**一律執行** `init_skill.py` 腳本。該腳本方便地生成一個新的技能模板目錄，自動包含技能所需的一切，讓技能建立過程更高效可靠。

用法：

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

腳本功能：

- 在指定路徑建立技能目錄
- 生成含有適當 frontmatter 和 TODO 占位符的 SKILL.md 範本
- 建立範例資源目錄：`scripts/`、`references/` 和 `assets/`
- 在每個目錄中新增可自訂或刪除的範例檔案

初始化後，視需要自訂或移除生成的 SKILL.md 和範例檔案。

### 步驟四：編輯技能

編輯（新生成或現有的）技能時，請記住技能是為另一個 Claude 實例使用而建立的。包含對 Claude 有益且非顯而易見的資訊。考慮哪些程序知識、特定領域細節或可重用素材，能幫助另一個 Claude 實例更有效地執行這些任務。

#### 學習成熟的設計模式

根據技能的需求查閱這些參考指南：

- **多步驟流程**：循序工作流程和條件邏輯見 references/workflows.md
- **特定輸出格式或品質標準**：範本和範例模式見 references/output-patterns.md

這些檔案包含有效技能設計的既定最佳實踐。

#### 從可重用技能內容開始

開始實作時，從上面識別的可重用資源入手：`scripts/`、`references/` 和 `assets/` 檔案。請注意此步驟可能需要使用者輸入。例如，實作 `brand-guidelines` 技能時，使用者可能需要提供品牌素材或範本存放在 `assets/` 中，或文件存放在 `references/` 中。

新增的腳本必須實際執行以確認無 bug 且輸出符合預期。若有許多類似腳本，只需測試具代表性的樣本，在確保信心和完成時間之間取得平衡。

任何不需要的範例檔案和目錄都應刪除。初始化腳本在 `scripts/`、`references/` 和 `assets/` 中建立範例檔案以示範結構，但大多數技能不需要全部。

#### 更新 SKILL.md

**撰寫指南：** 一律使用祈使/不定詞形式。

##### Frontmatter

撰寫含 `name` 和 `description` 的 YAML frontmatter：

- `name`：技能名稱
- `description`：這是技能的主要觸發機制，幫助 Claude 理解何時使用技能。
  - 同時包含技能的功能說明，以及使用時機/情境的具體觸發條件。
  - 將所有「使用時機」的資訊放在這裡——不放在 body 中。body 只在觸發後才載入，因此 body 中的「何時使用此技能」段落對 Claude 毫無幫助。
  - `docx` 技能的描述範例：「完整的文件建立、編輯與分析工具，支援追蹤變更、注釋、格式保留和文字擷取。當 Claude 需要處理專業文件（.docx 檔案）時使用，包含：(1) 建立新文件、(2) 修改或編輯內容、(3) 處理追蹤變更、(4) 新增注釋，或任何其他文件任務」

YAML frontmatter 中不要包含任何其他欄位。

##### Body

撰寫使用技能及其捆綁資源的指示。

### 步驟五：打包技能

技能開發完成後，必須打包成可發布的 .skill 檔案，與使用者共享。打包流程會先自動驗證技能，確保符合所有需求：

```bash
scripts/package_skill.py <path/to/skill-folder>
```

可選擇指定輸出目錄：

```bash
scripts/package_skill.py <path/to/skill-folder> ./dist
```

打包腳本將：

1. **驗證**技能，自動檢查：

   - YAML frontmatter 格式和必要欄位
   - 技能命名慣例和目錄結構
   - 描述完整性和品質
   - 檔案組織和資源參照

2. 驗證通過後**打包**技能，建立以技能命名的 .skill 檔案（例如 `my-skill.skill`），包含所有檔案並維持正確的目錄結構以供發布。.skill 檔案是副檔名為 .skill 的 zip 壓縮檔。

若驗證失敗，腳本會回報錯誤並退出，不建立套件。修正所有驗證錯誤後再次執行打包指令。

### 步驟六：迭代

測試技能後，使用者可能會要求改善。這通常發生在使用技能後不久，對技能表現有新鮮的記憶時。

**迭代工作流程：**

1. 在實際任務上使用技能
2. 注意困難或效率低落之處
3. 確認 SKILL.md 或捆綁資源應如何更新
4. 實作變更並再次測試
